## 摘要
大型语言模型（LLM）智能体日益依赖由共享提供商提供、由异构下游智能体访问的外部工具。现有方法通过参数更新、提示优化或智能体侧记忆在智能体端改善工具使用，使得工具知识难以共享，且局限于过去任务中观察到的行为。我们认为可重用的工具知识应由工具提供商维护。我们提出ToolAtlas，一个基于图的框架，通过执行验证探测，构建持久的提供商侧工具记忆，涵盖工具能力、失败边界和跨工具组合。推理时，智能体通过自适应图遍历查询工具记忆。在涵盖八个服务的两个基于MCP的基准测试中，ToolAtlas在pass@1和pass@4上分别比现有工具侧优化和智能体侧记忆基线提高多达21.61%和18.61%。同样的工具记忆还能在不同环境实例和智能体框架间迁移，无需重新训练或任务时探索，在pass@1/pass@4上分别获得高达24.16%/16.22%和17.49%/14.27%的相对增益。消融研究表明，这些增益源于工具中心记忆组织与能力引导的执行探测相结合。这些结果确立了提供商侧工具记忆作为工具服务器有效且可重用的范式。我们的代码见：https://github.com/PuppyKnightUniversity/ToolAtlas。

## Abstract
Large language model (LLM) agents increasingly rely on external tools served by shared providers and accessed by heterogeneous downstream agents. Existing approaches improve tool use on the agent side through parameter updates, prompt refinement, or agent-side memory, making tool knowledge difficult to share and limited to behaviors observed in past tasks. We argue that reusable tool knowledge should instead be maintained by the tool provider. We introduce ToolAtlas, a graph-based framework that builds a persistent provider-side tool memory of tool capabilities, failure boundaries, and cross-tool compositions through execution-verified probing. At inference time, agents query the tool memory via adaptive graph traversal. Across two MCP-based benchmarks spanning eight services, ToolAtlas outperforms existing tool-side optimization and agent-side memory baselines by up to 21.61% in pass@1 and 18.61% in pass@4. The same tool memory also transfers across environment instances and agent frameworks without retraining or task-time exploration, yielding up to 24.16%/16.22% and 17.49%/14.27% relative gains in pass@1/pass@4, respectively. Ablation studies show that these gains arise from combining tool-centered memory organization with capability-guided execution probing. These results establish provider-side tool memory as an effective and reusable paradigm for tool servers. Our code is in: https://github.com/PuppyKnightUniversity/ToolAtlas.

### 1. 论文的核心问题与整体含义

*   **研究背景**：随着模型上下文协议（MCP）等技术的发展，大型语言模型（LLM）智能体日益依赖共享的外部工具服务器来完成复杂任务。一个工具服务器可能被多个异构的下游智能体调用。
*   **核心问题**：现有提升LLM工具使用能力的工作主要集中在 **智能体侧（Agent-Side）**，例如通过微调模型参数、优化提示词或构建智能体侧记忆。这带来了两个结构性的**局限**：
    1.  **泛化困境（Generalization Dilemma）**：智能体A学到的工具使用经验（如记忆）与智能体A自身的交互模式、推理风格、环境假设强绑定，无法有效共享给智能体B，导致每个智能体都需要重新“发现”已知的工具行为。
    2.  **能力盲区（Capability Blindness）**：智能体侧记忆仅是其过往任务的被动记录，只覆盖了被特定任务触发的工具行为子集，无法主动、全面地揭示工具真实的**能力边界**（它能做什么/不能做什么）以及未被使用的**工具组合潜力**。
*   **整体含义**：论文提出了一种范式转变——**将工具使用知识的所有权从智能体转移到工具提供商**。核心思想是“一次学习，随处重用”，即在工具服务器端构建一个持久、可共享、可验证的工具能力图谱，供所有下游智能体查询使用，从而解决知识复用和覆盖不全的问题。

### 2. 论文提出的方法论

论文提出了 **ToolAtlas** 框架，其核心是在工具服务器端构建一个三层图结构记忆，并通过主动探测而非被动记录的方式填充和更新此记忆。

#### 核心思想
*   **提供者侧记忆**：工具知识作为记忆，与工具服务器而非智能体绑定，由提供商维护，实现知识跨智能体共享。
*   **执行验证与主动探索**：摆脱依赖历史任务的被动记录，通过主动生成探测任务并执行，验证工具的真实能力和边界，并将验证过的结果写入记忆。

#### 关键技术细节与流程

ToolAtlas 包含三个主要阶段：

1.  **记忆引导 (Memory Bootstrapping)**：
    *   **生成种子任务**：为服务中的每个工具，结合其技术文档和从网络上检索到的真实用法片段，合成一组初始的“任务-验证器”种子对。
    *   **多轨迹执行**：让智能体执行种子任务，产生多组独立的执行轨迹（成功与失败）。
    *   **记忆归纳 (Memory Induction)**：将智能体产生的特定轨迹，蒸馏成去中心化、与智能体无关的图结构记忆。它构建了三层图：
        *   **工具-轨迹图 (Gtrace)**：节点是去除智能体特定语法的“智能体中立轨迹”，保留了核心的工具调用序列和意图。
        *   **工具-能力图 (Gcap)**：节点是工具本身，每个工具节点下积累了三类条目：**能力 (Affordance, U)** （工具能可靠做什么）、**边界 (Boundary, B)** （工具在何种情况下失效）和**协同使用 (Co-usage, C)**（工具常与哪些工具组合使用）。
        *   **工具-策略图 (Gstrat)**：节点是从多个任务中提炼出的、更高层次的通用规划策略（如“先进行工作区和约束发现，再执行操作”）。

2.  **能力探索 (Capability Exploration)**：
    *   这是一个迭代的闭环过程，专门用于解决“能力盲区”。
    *   **子图采样**：从工具-能力图 `Gcap` 中采样一个由相关工具组成的局部子图。
    *   **生成探测任务**：根据子图当前已知的能力与边界，通过LLM沿两个互补方向生成探测任务：
        *   **向外探索 (Boundary)**：测试工具可靠性的极限（例如，要求一个只能列出浅层文件大小的工具去递归计算深层文件夹大小）。
        *   **向内探索 (Affordance)**：挖掘未被记录的潜在能力和工具组合。
    *   **执行与回写**：新生成的探测任务在真实环境中执行并验证，结果再次通过“记忆归纳”写回三层图中，丰富和完善图结构。

3.  **动态记忆遍历 (Dynamic Memory Traversal)**：
    *   推理时，智能体面对一个新任务，一个轻量级的导航器会在 `MT` 图上进行自适应图遍历。
    *   **操作**：导航器通过 `ReadTrace`（读取轨迹）、`Expand`（扩展至邻居节点）、`ReadTool`（读取工具能力/边界条目）和 `ReadStrategy`（读取策略）四种操作，从记忆图中收集相关信息。
    *   **生成指导**：遍历结束时，收集到的上下文被压缩成任务相关的指导，注入到工作智能体的提示上下文中，引导其做出更好的工具使用决策。

### 6. 论文的主要结论与发现

*   **范式有效性**：提供者侧工具记忆是一种有效且可重用的范式。ToolAtlas 在各种设置下均显著优于工具侧优化和智能体侧记忆基线。
*   **性能提升显著**：在 MCPMark 上，pass@1 最高提升 **21.61%**，pass@4 最高提升 **18.61%**。在更弱的 LLM 骨干上（如 GPT-5.4-mini），提升幅度更大，表明它能弥补推理能力的不足。
*   **强大的泛化能力**：
    *   **跨环境**：记忆可从一套环境实例（如一个GitHub仓库）迁移到另一套，无需任何修改，相对提升高达 24.16%。
    *   **跨智能体**：由一种智能体构建的记忆，可以直接被行动模式完全不同的另一种智能体使用，并带来显著增益。
*   **关键贡献来自于设计耦合**：消融实验证实，性能的大幅提升来源于**将工具中心的记忆组织（Gcap）与能力引导的执行探测（Frontier Exploration）相结合**。其中，对工具能力的主动探索（而非被动记录）是主要的知识来源。

### 7. 优点

*   **范式创新性强**：挑战了主流的智能体侧优化思路，提出“工具侧记忆”这一新的视角，从根本上解决了工具知识的共享壁垒问题。
*   **设计精巧且系统**：三层图结构的设计清晰地将工具执行的轨迹、工具本身的能力/边界、以及高层次的规划策略解耦并关联起来，结构完整。主动的“边界/能力”探索机制巧妙地解决了被动记忆的覆盖盲区。
*   **实验论证充分、扎实**：实验设计考虑周全，不仅证明了“更好”，还通过跨环境、跨框架和详细的消融实验，清晰论证了其方法的“泛化性”和“为什么更好”，并提供了效用-成本分析，证据链完整。
*   **实用价值高**：“一次构建，处处使用”的模式非常契合当下 MCP 服务器生态的实际部署场景，降低了下游智能体的集成成本，让工具提供商可以主动优化工具的被使用能力。

### 8. 不足与局限

*   **评测环境相对静态**：实验基于公开基准测试，任务和工具API相对固定。论文自身也指出，未评测高度动态变化或对专业领域有严格限制的API，尚不清楚该方法在快速变化的真实生产环境中的表现。
*   **生命周期与维护成本未充分讨论**：论文主要关注任务成功率，对工具侧记忆的完整生命周期管理探讨不足。例如，当后端API更新、功能变化时，记忆的**重新验证、刷新和治理**机制需要额外资源，这部分成本未计入评估。
*   **探索过程的成本与风险**：能力探索阶段需要主动生成并执行大量探测任务，这在真实环境中可能引入不可预见的副作用（如误操作数据库）或产生显著的 API 调用费用，这部分离线构建成本在论文的核心指标对比中未作为主要维度。
*   **依赖 LLM 的能力**：整个框架，包括任务生成、记忆归纳、图遍历等，都强依赖底层 LLM 的推理和代码生成能力。在较弱的模型上可能无法有效地进行能力探索或归纳出有用的知识。
