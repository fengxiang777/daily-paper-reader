## 摘要
大语言模型代理可以通过积累经验得到改进，但自由形式的文本记忆随着交互次数的增长变得难以维护、验证和重用。近期的符号化方法学习可执行技能或程序化世界模型，但往往只存储局部过程或假设简化的动态。我们提出以对象为中心的环境建模（OCM），将经验组织成一个可执行的以对象为中心的环境模型。OCM维护两个相互关联的代码库：对象知识，用Python类定义环境实体和机制；过程知识，记录必须导入并使用对象模型的可重用交互模式。OCM以在线方式工作：在每个回合结束后，OCM反思轨迹，更新两个知识库，并验证所有过程都能在更新后的对象模型上执行。在后续交互中，代理采用渐进式知识揭露策略，先查看紧凑的代码签名，仅在需要时才阅读源代码。实验表明，OCM在多个基准测试中取得了最佳平均排名，并减少了无效动作，证明代理能够从构建以对象为中心的环境模型中获益。

## Abstract
Large language model (LLM) agents can improve through accumulated experience, but free-form textual memories become difficult to maintain, validate, and reuse as interactions grow. Recent symbolic approaches learn executable skills or programmatic world models, yet often store local procedures or assume simplified dynamics. We propose Object-Centric Environment Modeling (OCM), which organizes experience into an executable object-centric environment model. OCM maintains two connected code bases: object knowledge, which defines environment entities and mechanisms as Python classes, and procedure knowledge, which records reusable interaction patterns that must import and use the object model. OCM works in an online setting: after each episode, OCM reflects on the trajectory, updates both knowledge bases, and verifies that all procedures execute against the updated object model. During future interaction, the agent uses progressive knowledge disclosure to inspect compact code signatures first and read source code only when needed. Experiments show that OCM achieves the best average rank across benchmarks and reduces invalid actions, demonstrating that agents can benefit from building object-centric environment models.

### 1. 论文的核心问题与整体含义
现代大语言模型（LLM）智能体依赖经验积累来应对长周期、开放式的复杂任务。然而，现有的两类主流经验留存方式均存在结构性缺陷：
-   **自由文本记忆**：在交互增多后急剧膨胀，变得冗余、自相矛盾且难以审计，无法保证知识的一致性和有效性。
-   **符号化/程序化模型**：虽然提升了知识的可解释性，但往往走向另一个极端，要么仅学习局部的、孤立的“技能脚本”，要么试图构建覆盖所有状态转移的“一体化”世界模型，这在动态、部分可观测的环境中适用性差、工程代价高昂。

本文发现，真正的缺失在于一个处于中间地带的组织框架：**既能保持技能之上的整体一致性，又无需构建沉重的全能转移模型**。因此，作者提出了**以对象为中心的环境建模（OCM）**，其核心思想是将智能体的交互经验组织为一个以持久性环境实体（对象）为锚点的、可执行的共享代码模型，从而在保证结构化和可验证性的同时，实现知识的模块化、可复用与一致性增长。

### 2. 论文提出的方法论
OCM 的设计哲学是“过程知识必须根植于对象知识”，通过一个在线的“反思-更新-验证”闭环，将一个环境抽象为两个相互关联的 Python 代码库，并通过渐进式揭露机制辅助决策。

-   **核心思想**：
    -   **对象知识（`Ko`）**：定义环境中的实体（如容器、工具、材料）及其状态、可供性、约束和转变机制，表现形式为 Python 类。
    -   **过程知识（`Kp`）**：记录可复用的交互模式、规则、失败案例或启发式方法。**关键约束**是，每个过程文件**必须**显式导入并使用 `Ko` 中的类或函数，而不能是独立孤立的文本脚本。

-   **技术流程**：
    1.  **在线决策（渐进式知识揭露）**：
        -   为避免将全部代码塞入上下文窗口，智能体首先获取 `Ko` 的类/方法签名和 `Kp` 的函数索引。
        -   当决策存在不确定性时，它可以发出内部的 `Inspect[...]` 指令，精确检索相关对象的源码，实现按需加载。
    2.  **经验反思（Object-Centric Reflection）**：
        -   每个回合结束后，不直接对原始轨迹进行编码。模型先基于轨迹、奖励、当前签名和索引，生成一个**结构化的更新计划**（`P^k`）。
        -   该计划过滤掉暂时性的、任务特定的细节，明确要更新/创建的对象机制和过程模式，并标注需查阅的现有相关源码。
    3.  **知识更新与验证（Knowledge Update & Verification）**：
        -   根据更新计划生成候选代码。
        -   **可执行性验证**：一个至关重要的步骤。系统会强制编译并运行所有过程文件，以确保它们在更新后的对象知识库上能成功执行。验证公式为 `Verify(˜Ko, ˜Kp) = ∧ p∈˜Kp Exec(p | ˜Ko)`。未通过验证的更新将被驳回和修正，从而阻止不一致的知识污染模型。
    4.  **知识合并**：通过验证的 `∆Co` 和 `∆Cp` 被正式合入知识库。
